## 1. 写这份文档的目的

这份文档不是为了重复 `P0-P3` 的已有结论，而是为了记录一次后续结构改造实验为什么失败。

这次失败的价值在于:

1. 它说明 `P0-P3` 后面的问题不在“方向完全错了”
2. 它也说明“知道下一步该往哪改”不等于“实现时自然不会踩坑”
3. 它把哪些改法会破坏当前基线、哪些边界必须守住，暴露得很清楚

后续如果继续推进下一轮改造，应该先读这份文档，避免重复走同一类弯路。

## 3. 这次实验我实际做了哪些改动

这轮实验不是小修小补，而是一次偏结构性的推进。主要做了四类改动:

### 3.1 改动一: 高分主线改成 `Tracked` 与 `Lost` 分开处理

原本 `Step2` 是:

- `Tracked + Lost` 一起进入高分 IoU 主匹配池

实验版改成:

- 先只让 `Tracked` 走主线高分匹配
- 再把 `Lost` 放到单独的高分恢复步骤里处理

设计意图:

- 不让 `Lost` 再完全依附于 `Tracked` 的纯 IoU 命运
- 把旧 ID 恢复能力前移

实际问题:

- 这已经不再是“在主干后面加一个受限恢复层”
- 而是在改写 ByteTrack 最稳定的高分主线结构


### 3.2 改动二: 引入 `short-lost / mid-lost` 生命周期分层

实验版新增了:

- `short_lost_max_frames`
- `mid_lost_max_frames`
- `lost_freeze_frames`

设计意图:

- 把中短时丢失区间显式拆开
- 避免 zombie 参数和 lost 参数混用

实际问题:

- 生命周期分层本身不是错的
- 但我把它和主线重构同时推进，导致结果很难归因
- 也让“恢复能力前移”与“主干被扰动”混在一起发生


### 3.3 改动三: 给 unmatched high 加了 `birth_score / recover_score` 统一决策 v1

实验版不再只做简单硬路由，而是比较:

- `birth_score`
- `recover_score`

然后决定:

- 新生
- 认旧
- 延后

设计意图:

- 让 `birth` 和 `recover` 在同一决策面上比较

实际问题:

- 统一打分的落点太早、影响面太大
- 它不再只是补充 `unmatched high` 的歧义判别
- 而是开始重塑高分决策面本身


### 3.4 改动四: 给 low-score 增加了 `recent/mid-lost` 窄恢复支路

实验版给低分检测增加了一轮受限恢复:

- 仅对 `recent/mid-lost`
- 使用 stricter motion/shape/appearance gate

设计意图:

- 降低 `07/13` 这类 recall-heavy 序列的无谓 FN

实际问题:

- 这条支路单独看并不一定错误
- 但它和前面三项结构改造叠加后，会把“追回来一些框”的收益与“主线变脏”的副作用混在一起

## 6. 这次失败的根本原因

### 6.1 不是文档方向错，而是实现越界了

文档的主判断是:

- 不要继续主要围绕 zombie 微调
- 恢复能力要前移
- 但不要破坏 ByteTrack 高分主线稳定性

这次失败说明:

- 文档的方向判断没有错
- 真正错的是我实现时没有守住“前移但不扰动主线”的边界


### 6.2 我没有严格按照文档的实验纪律来做

文档强调:

- 一轮只改一个主要结构因素
- 先重点序列，再全序列

我实际做的是:

1. 高分主线结构改造
2. 生命周期分层
3. unmatched-high 统一打分
4. low-score recovery

这些一起上。

后果是:

- 收益和副作用全耦在一起
- 一旦结果恶化，很难做因果切分


### 6.3 最关键的错误是把 `Lost` 恢复拉得过宽、过早

真正的问题不是“恢复前移”这四个字本身，而是:

- 我把 `Lost` 更深地拉回了高分主决策面
- 让它不再只是一个受限补救对象
- 而开始影响主线关联的定型方式

这直接导致:

- `FP` 增长
- `IDSW` 增长
- `Frag` 暴涨


### 6.4 当前系统的主矛盾仍然是“先切碎，再补救”

这次实验没有解决这个问题，反而放大了它。

表现为:

- 追回来一些框
- 但不是更连续地保住同一个 ID
- 于是 headline 里某些召回相关数字看起来不坏
- 但 hardest sequences 的身份质量明显恶化


## 7. 这次失败至少带来了哪些确定经验

### 7.1 经验一: 不能在现有基线上直接大改高分主线

`P0-P3` 之后最需要守住的是:

- 高分主线稳定性

任何“把 Lost 从主线里抽出来再重接一套高分恢复”的做法，都必须极端谨慎。

至少从这次实验看:

- 直接这样改，风险非常高


### 7.2 经验二: `recover vs birth` 的统一打分不能越过主线边界

统一打分不是不能做，而是只能做在:

- 少量 `unmatched high` 的歧义 case 上

不能做成:

- 广泛改变高分主线的隐式替代机制


### 7.3 经验三: `low-score recovery` 不能和主线结构重写一起上

这条支路后续仍可能有价值。

但正确顺序应该是:

1. 先守住高分主线
2. 再单独验证 low-score recovery 对 `07/13` 的帮助

而不是同时上。


### 7.4 经验四: “追回来更多”不等于“做对了”

如果一个改动带来了:

- `FN` 下降

同时又带来了:

- `FP` 明显上升
- `Frag` 明显上升
- `IDSW` 明显上升

那么它大概率不是“更好地跟踪”，而是“更激进地补救”。

在 fixed-scene MOT 里，这种收益要高度警惕。


### 7.5 经验五: 文档里的边界语句必须当成实现约束，而不是仅仅方向提示

这次最大的教训之一是:

- 文档里“不要破坏主线”“一轮只改一个主要结构因素”“不要让 ReID/恢复逻辑过宽介入主干”

这些不是建议语气，而应该当成硬约束。


## 8. 这次失败后，正确的做法是什么

### 8.1 结论一: 当前实验分支不应继续累计开发

原因:

- 两轮全序列结果都已经证明方向落点不对
- 再继续在这条分支上做局部修补，收益大概率很小


### 8.2 结论二: 必须回到 `P0-P3` 基线继续下一轮实验

当前已经确认:

- git `HEAD` 对应 `P0-P3` 基线
- 复跑结果目录 `..._bytetrack_5` 已经重新固化

后续所有实验，都应该以这份基线为唯一对照。


### 8.3 结论三: 下一轮只能做更窄的 `R1-lite`

下一轮如果继续推进，建议只做:

- 保留 `Step1/Step2` 主体不变
- 只对主线之后留下的少量 `unmatched high` 做受限 `recent/mid-lost` 恢复
- 不让整池 `Lost` 重新进入高分主决策面
- 不同时叠加 low-score recovery
- 不同时叠加新的 unified scoring 重构


## 9. 应该明确记录的责任判断

这次失败的责任判断应该写清楚:

1. 不是 `P0-P3` 文档方向错
2. 不是“fixed-scene 特化本身走不通”
3. 主要是我在实现时没有严格按照文档边界来推进
4. 我把结构改动推进得过快、过宽、过多
5. 结果是把本来应该受保护的主线稳定性打坏了

这点必须明确，否则后续很容易错误地把失败归因到文档判断本身。


## 10. 后续使用建议

这份文档建议作为后续结构实验的前置阅读材料。

尤其在再做下一轮改动前，应先确认以下三点:

1. 这次改动是否真的不动高分主线主体
2. 这次改动是否只引入一个主要结构因素
3. 这次改动是否先在重点序列验证，再上全序列

如果有任意一条不满足，就很可能再次走到类似失败里。
