## 1. 目标与当前现实

### 1.1 目标指标

当前希望在 SOMPT22 上尽量逼近并稳定达到:

- `HOTA >= 54`
- `MOTA >= 67`
- `IDF1 >= 67`
- `IDSW < 500`
- `Association FPS > 100`

### 1.2 当前最新已验证结果

为了避免把“数据缓存是否重建”和“算法逻辑是否变动”混在一起，这里保留两组结果:

#### 1.2.1 旧 fresh baseline（完整重跑 dets + embs + tracking）

- `HOTA = 53.023`
- `MOTA = 65.710`
- `IDF1 = 66.179`
- `IDSW = 815`
- `Association FPS = 18.9`

对应结果目录:

- `runs_fullrerun_20260315_094907/`

#### 1.2.2 当前最新全序列结果（P0-P3 完成后，复用已有 dets + embs，仅重跑 tracking/eval）

- `HOTA = 53.549`
- `MOTA = 66.013`
- `IDF1 = 67.014`
- `IDSW = 747`
- `Association FPS = 28.3`

对应结果目录:

- `runs_reusecache_eval_20260316_043347/`

对应汇总文件:

- `runs_reusecache_eval_20260316_043347/mot/yolov8m_pretrain_crowdhuman_osnet_x0_25_msmt17_bytetrack/person_summary.txt`

这组结果说明:

- `P0-P3` 的方向是有效的，五个 headline 指标都比旧基线更好
- `IDF1` 已经越过 `67`
- 但 `HOTA 54`、`MOTA 67`、`IDSW < 500`、`Association FPS > 100` 仍未达到
- 因而系统已经进入“方向验证通过，但离最终目标还有明显工程距离”的阶段

### 1.3 这组结果真正说明了什么

结合上面两组结果，可以确认当前系统并没有崩:

- 主干 tracking 是能工作的
- `P0-P3` 改造后指标仍在向目标边缘推进
- 方案方向并非完全错误

但它也说明系统离目标仍有明显距离，而且这个距离并不是靠一两个参数微调就能补齐的。

最重要的现实判断有两条:

#### 1.3.1 `IDSW` 确实高，但不是唯一问题

当前 `IDSW = 815` 明显过高，直接拖累:

- `IDF1`
- `HOTA`
- `Frag`
- `IDs`

这意味着身份连续性是主要矛盾之一。

#### 1.3.2 `MOTA 67` 不可能靠单独降低 `IDSW` 达成

当前:

- `CLR_FN = 137202`
- `CLR_FP = 45746`
- `IDSW = 815`
- `GT_Dets = 535904`

MOTA 的主要损失项仍然是:

- `FN`
- `FP`

而不是 `IDSW`。

即使把 `IDSW` 从 `815` 直接降到 `500`，理论上对 MOTA 的提升也只有约:

- `(815 - 500) / 535904 * 100 ≈ 0.059`

也就是说，MOTA 只能从 `65.71` 涨到大约 `65.77`，远达不到 `67`。

因此，后续整改必须同时做两件事:

1. 改善身份保持，降低 `IDSW / Frag / IDs`
2. 改善召回与错误新生控制，降低 `FN / FP`


## 2. 当前算法到底是什么

当前实现已经不是原始 ByteTrack，而是一个四层叠加系统:

### 2.1 主干关联层

保留原始 ByteTrack 两段式主干:

1. 高分检测与 `tracked + lost` 做 IoU 关联
2. 低分检测与剩余 `tracked` 做第二轮 IoU 关联

这一层负责大多数短时稳定跟踪。

### 2.2 生命周期控制层

新增:

- `entry zone`
- `exit zone`
- `birth_confirm_frames`
- `birth suppression`
- `adaptive effective zone`

这一层在决定:

- 什么检测可以新生
- 什么轨迹算离场
- 什么未匹配检测应优先当作旧 ID 回归

### 2.3 长时记忆恢复层

新增:

- `lost -> zombie`
- `frozen_mean`
- `zombie rescue`
- ReID 仅在 zombie 阶段生效

这一层试图恢复原始 ByteTrack 会彻底丢掉的长时断链身份。

### 2.4 在线空间先验层

新增:

- `SpatialPriorField`
- `support_count`
- `birth_count`
- 在线构建 `entry/core masks`

这一层试图从固定监控场景的长时稳定统计中学习:

- 人通常从哪里出现
- 哪些区域是长期核心区域


## 3. 总体结论

先给最终结论，不兜圈子。

### 3.1 大方向没有根本性错误

当前系统的三条主线本身是合理的:

1. 不轻易破坏 ByteTrack 主干
2. 把 fixed-scene 先验显式放进生命周期
3. 把外观信息优先限制在高风险阶段

所以，当前离目标较远，不能简单归结为:

- “fixed-scene 特化方向是错的”
- “spatial prior 不该做”
- “ReID 就该彻底塞进主干”

这些简单结论都不准确。

### 3.2 真正的问题不在单个参数，而在系统控制流

当前最大的根因不是某一个阈值错了，而是整套系统没有形成真正自洽的控制逻辑。

更具体地说，当前系统存在三类核心问题:

1. 明确的状态机实现 bug
2. 身份恢复插入时机过晚
3. 生命周期门控与空间先验之间的职责边界不清

一句话概括当前版本:

- 系统经常先把轨迹切碎
- 然后再试图用一个很晚、很窄、很苛刻的 zombie rescue 通道去补救
- 而这条补救通道自己又受到状态 bug 和硬门控影响
- 于是很多本来应该保住的 ID 最终还是碎掉了

### 3.3 如果继续沿当前思路推进，真正该保留的是什么

应该保留的是:

- ByteTrack 主干框架
- fixed-scene 生命周期特化思路
- 空间先验的长期统计思想
- ReID 不应无脑接管全局主干的谨慎态度

真正该改变的是:

- 恢复链路的位置
- 生命周期状态维护方式
- birth/exit/prior 的耦合方式


## 4. 已确认的硬问题

下面这些不是“可能存在”，而是已经被代码直接证实的问题。

### 4.1 硬问题一: `exit zone` 使用了陈旧框

修复状态: 已修复（2026-03-16，代码与单元测试已补齐）

#### 4.1.1 问题现象

`STrack.tlwh` 在初始化时被赋值，但之后正常 `update()` 和 `re_activate()` 并不会刷新它。

轨迹真正的当前位置来自:

- `mean`
- `covariance`
- `xyxy`
- `get_tlwh_for_matching()`

但在轨迹从 `Tracked -> Lost` 的那一刻，`exit zone` 判定用的是:

- `track.tlwh`

而不是当前 KF 位置。

#### 4.1.2 这为什么严重

这会制造一种非常致命的错误:

1. 目标从边缘进入
2. 后续已经走到画面中部
3. 发生遮挡或短时丢失
4. 代码仍用首帧附近的旧 `tlwh` 去判断是否在 exit zone
5. 结果把一个中部遮挡误判成“边缘离场”

随后旧轨迹会进入:

- `exit_pending`
- `grace`
- `remove`

从而失去正常恢复机会。

#### 4.1.3 对指标的影响

这类错误通常会同时造成:

- `IDSW` 上升
- `Frag` 上升
- `IDs` 上升
- `IDF1` 下降
- `HOTA` 下降

这是典型的高伤害状态机错误，不是边缘噪声。

#### 4.1.4 根本原因

`STrack` 当前事实上同时存在两种框语义:

- 初始化检测框: `tlwh`
- 当前估计框: `mean -> xyxy / matching tlwh`

但生命周期判定没有统一强制使用“当前状态框”。

#### 4.1.5 结论

`exit zone` 逻辑目前不可信，必须优先修复。

### 4.2 硬问题二: 重激活后没有清空 `lost_frame_id` 和 `frozen_mean`

修复状态: 已修复（2026-03-16，代码与单元测试已补齐）

#### 4.2.1 问题现象

当前 `re_activate()` 和 `update()` 不会清理:

- `lost_frame_id`
- `frozen_mean`

这意味着:

1. 一条轨迹曾经 lost 过
2. 它后来成功重新跟踪
3. 但内部仍保留上一次丢失时的 frozen 状态
4. 下一次再丢时，仍可能使用旧冻结位置参与后续门控和匹配

#### 4.2.2 这为什么严重

`frozen_mean` 的语义本来应该是:

- “本次丢失过程中，在超过预测窗口后冻结下来的位置”

它不应该跨越一次完整的成功重跟踪继续存活。

否则会导致:

- 空间门控逐渐偏离当前真实位置
- 真正该匹配的检测被拒绝
- 同一目标经历多次遮挡后越来越难接回

#### 4.2.3 对指标的影响

这个问题对“反复遮挡、反复重现”的目标伤害特别大，会系统性放大:

- `IDSW`
- `Frag`
- `IDs`

这类问题往往集中出现在复杂交互序列里，而当前 hardest sequence 正符合这个特征。

#### 4.2.4 结论

这也是必须优先修复的 P0 级问题。

### 4.3 硬问题三: 当前测试没有兜住真正最危险的状态机错误

修复状态: 已修复（2026-03-16，已补 `exit zone` 当前框读取、`re_activate/update` 状态清理、跨周期 frozen 重建测试）

当前测试覆盖了不少模块:

- zombie ReID
- birth confirm
- exit zone 延迟删除
- spatial prior region
- adaptive zone

但它们主要在保护“当前期望行为”。

测试目前没有直接兜住以下关键行为:

1. `exit zone` 必须基于当前轨迹位置而非陈旧 `tlwh`
2. `re_activate()` 后必须清空 lost/zombie 残留状态
3. 一条轨迹多次失而复得后，其 frozen 状态必须按每次 lost 独立重建

也就是说，测试并不等于状态机是干净的。


## 5. 当前最根本的结构性问题

这一部分比单点 bug 更重要，因为它决定了系统上限。

### 5.1 问题一: 身份恢复插入得太晚

这是当前版本最核心的结构性问题。

当前 ReID 只在:

- Step4
- unmatched high detections
- center-zone
- zombie tracks

这一条非常窄的路径上生效。

这意味着大量真正影响 `IDSW` 的 case 根本没有进入这条恢复链路。

典型包括:

- 5 到 20 帧中短时遮挡
- 邻近目标交叉
- 人群中的短时漂移
- 低分续命后再次稳定
- lost 状态下仍然有机会被救回的目标

这些错误往往在进入 zombie 阶段之前就已经发生了。

因此，单独继续增强 final zombie rescue，收益一定有限。

#### 结论

当前系统最大的上限瓶颈不是 zombie rescue 太弱，而是:

- 真正需要恢复的 case，大量发生在 zombie 之前

### 5.2 问题二: 当前结构是“硬路由 + 晚补救”，而不是“统一打分 + 生命周期偏置”

当前控制流大致是:

1. 先用主干 IoU 做完高分关联
2. 再做低分补关联
3. 没接上就先 lost
4. 没进入 entry 的 unmatched high，再很晚地尝试 zombie rescue
5. 最后没救回来就新生或删除

这个结构的根本问题在于:

- 各个模块是串联抢控制权
- 不是在统一候选空间内共同决策

结果就是:

- 有些地方过于保守，造成漏新生和漏接回
- 有些地方又过早放行新生，造成重复 ID
- 有些地方本应优先认回旧 ID，却在 earlier stage 被路由到新生逻辑

#### 结论

当前系统不是“逻辑细化版 ByteTrack”，而更像“多个局部规则的串接版 ByteTrack”。

### 5.3 问题三: 空间先验承担了过多职责

当前 `spatial prior` 既想做:

- entry/core 划分
- center override
- 几何 entry 的替代
- 长时场景结构学习

但它的数据来源是:

- tracker 自己在线确认过的 support/birth 事件

这引入了一个根本风险:

- 如果 tracker 当前已经有错误新生、错误复活、错误删轨
- 这些错误会被写回 spatial field
- 然后 spatial field 再反过来影响后续新生/恢复逻辑

这实际上是一个自我强化闭环。

#### 结论

空间先验本身方向对，但当前实现需要防污染机制，否则很容易把 tracker 的错误固化成“场景规律”。

### 5.4 问题四: `adaptive effective zone` 目前更像弱辅助，而不是强先验

当前默认是:

- `always_expand`
- `all_high`
- 单调扩张

这使得 effective zone 在长序列里很容易逐步变大，最终退化成:

- 近似整图有效

一旦退化到这个状态，它对 entry gating 的实质约束就会明显减弱。

它仍然有价值:

- 早期 warmup
- outside-before-expand 放行

但它现在并不是一个足够强、足够稳定的场景边界建模器。

#### 结论

`adaptive effective zone` 不是当前指标最主要的伤害来源，但它也不是当前系统真正可靠的核心控制器。


## 6. 对各模块设计方向的评判

这里区分“方向对不对”和“实现得好不好”。

### 6.1 ByteTrack 主干

#### 方向判断

保留原始 ByteTrack 主干是对的。

#### 原因

- 原始 ByteTrack 的优势本来就在两阶段 IoU 主干的稳定性
- 一上来全面改主干，回归风险很大
- 当前系统已经证明主干本身并没有崩

#### 但需要明确

“保留主干”不等于:

- 不能对 `lost` 阶段加局部恢复增强
- 不能对 ambiguous case 增加 appearance-assisted tie-break

真正不该做的是:

- 让 ReID 粗暴接管全局主关联

而不是:

- 完全把 ReID 隔离到 zombie 之后

### 6.2 Birth control

#### 方向判断

新生控制方向是对的，但当前实现不够统一。

#### 做对的地方

- `new_track_thresh`
- `birth_confirm_frames`
- duplicate suppression

这些都是减少错误新生和重复 ID 的合理手段。

#### 做得不够好的地方

当前新生控制是:

- entry gate
- strict gate
- outside-before-expand 放行
- center unmatched fallback
- spatial prior override

多套逻辑并行作用，导致:

- 规则解释成本高
- 行为边界不清
- 某些 case 过硬
- 某些 case 又放得太快

#### 结论

birth control 应该保留，但需要重构成一套统一决策逻辑。

### 6.3 Exit control

#### 方向判断

在 fixed-scene 里做 exit zone 是合理的。

#### 原因

- 边缘附近消失确实更可能是真离场
- 给边缘离场目标单独 grace/remove 机制是合理工程优化

#### 当前主要问题

不是 exit 思路错，而是:

- 它目前建立在错误的状态读取上
- 并且一旦误触发，代价太高

#### 结论

exit zone 不应删除，但必须:

1. 修复状态读取
2. 降低误判代价
3. 避免过早切断正常恢复链

### 6.4 Zombie rescue

#### 方向判断

zombie rescue 本身是对的。

#### 原因

- 原始 ByteTrack 对长时断链恢复能力弱
- fixed-camera 下长时遮挡后重现确实存在可利用规律
- ReID 放在高风险低频阶段，工程上更稳

#### 但必须明确

zombie rescue 不是主战场。

当前 hardest case 的大量损失发生在:

- zombie 之前

所以 zombie rescue 只能是补充，不应该被当成整个系统达标的主力杠杆。

#### 结论

保留 zombie rescue，但不要继续把主要研发资源压在它上面。

### 6.5 Spatial prior

#### 方向判断

方向是对的，而且有长期潜力。

#### 原因

- fixed-camera 下 entry region 本来就不一定等于图像边缘
- 中心区域也可能存在长期重现热点
- 用统计场替代几何边缘带，从理论上更符合场景规律

#### 当前问题

最大问题不是速度，而是:

- prior 来自 tracker 自己的输出
- tracker 自己的错误会污染 prior
- prior 又反过来影响后续 tracker

#### 结论

spatial prior 应继续保留，但在未加防污染机制前，不应被赋予过强的决策权。


## 7. 指标与分序列形态分析

### 7.1 当前问题不是单一模式

从分序列结果看，当前失败模式至少分两类。

### 7.2 类型 A: 以身份连续性问题为主

典型序列:

- `SOMPT22-08`
- `SOMPT22-10`

它们的共同特点:

- `IDSW` 很高
- `Frag` 很高
- `IDs` 偏高

尤其:

- `SOMPT22-08: IDSW = 245`
- `SOMPT22-10: IDSW = 187`

二者合计:

- `432 / 815 = 53.0%`

也就是说，全局超过一半的 IDSW 由这两个序列贡献。

#### 这通常意味着什么

这更像:

- 中短时遮挡没接回
- 近邻交叉中 ID 被切碎
- 被过早解释为离场
- 中心重现后没能认回旧轨迹
- 最终重复新生

这与“恢复链路太晚”和“状态机 bug 放大碎裂”的结论高度一致。

### 7.3 类型 B: 以召回问题为主

典型序列:

- `SOMPT22-07`
- `SOMPT22-13`

它们的特点是:

- MOTA 明显偏低
- IDSW 不是主导项
- 更像 recall 和 lifecycle control 没处理好

特别是:

- `SOMPT22-07: MOTA = 48.13`
- `SOMPT22-13: MOTA = 45.21`

这说明系统还有一部分大问题在于:

- 新目标没有及时建立
- 中等置信度回归没有保住
- 某些 track 被过早失活或删除

### 7.4 当前改法为什么没法同时打掉两类问题

因为当前系统的主要增强点是:

- final zombie rescue

而两类问题分别需要的是:

- 类型 A: 更早的身份恢复
- 类型 B: 更合理的召回与新生控制

也就是说，当前重点增强的位置，并不是最需要增强的位置。


## 8. 根本原因归纳

综合文档、代码和结果，我认为当前距目标较远的根本原因可以归结为四条。

### 8.1 根因一: 系统把太多可恢复错误留到了 zombie 之后

这是第一根因。

中短时遮挡、交叉漂移、本地失配，本应在:

- `tracked/lost` 阶段

就被尽量恢复。

但当前系统主要把外观恢复放在:

- `zombie + center unmatched high`

于是大量身份错误在进入 zombie 之前就已经定型。

### 8.2 根因二: 生命周期状态维护不干净

这是第二根因。

当前状态机存在:

- 位置语义不统一
- lost/zombie 残留状态跨周期污染
- exit_pending 误触发代价过大

这种问题会直接放大 ID 碎裂，不是小修小补能盖过去的。

### 8.3 根因三: 生命周期门控过硬，但又不敢真的硬到底

这是第三根因。

文档上想表达的是:

- 中心非新生
- 优先认旧

但 tuned config 又把:

- `strict_entry_gate = false`

说明系统实际上无法依赖这套硬门控稳定运行，否则 recall 会掉。

也就是说，当前门控处于一种尴尬状态:

- 不够软，容易误伤
- 又不够硬，无法真正形成稳定规则

### 8.4 根因四: 空间先验尚未和生命周期系统形成正确分工

这是第四根因。

当前 spatial prior 既想学结构，又想参与实时决策。

但在 tracker 本身还不够稳定的阶段，它更适合作为:

- 偏置项
- 参考项
- 风险修正项

而不应成为:

- 强硬的主判断来源


## 9. 最终整改原则

如果要真正推进到目标区间，后续必须遵循下面这些原则。

### 9.1 原则一: 先修状态机，再谈高层策略

在以下问题修掉之前，不应继续做大规模参数调优:

1. `exit zone` 使用陈旧框
2. `re_activate/update` 不清理 lost/zombie 残留状态
3. 当前框语义没有统一

否则任何调参结果都不稳定。

### 9.2 原则二: 恢复前移，而不是一味增强 zombie

后续最应该加强的是:

- `lost` 阶段的恢复
- 中短时遮挡恢复
- ambiguous unmatched high 的局部 appearance-assisted recovery

而不是继续把大部分资源投入到 final zombie rescue。

### 9.3 原则三: Birth/exit/prior 必须形成统一控制逻辑

后续应该避免继续在流程里堆叠更多 if/else 局部规则。

需要统一回答的问题是:

- 一个未匹配检测，为什么被解释成“新生”
- 为什么被解释成“旧 ID 回归”
- 为什么某条轨迹被解释成“离场”

这些决策要形成统一语义，而不是靠多个局部模块串联抢控制权。

### 9.4 原则四: Spatial prior 在 tracker 稳定前只能是偏置，不应是主裁判

在没有防污染机制之前:

- spatial prior 应降低权重
- 不应直接强制覆盖主生命周期逻辑

否则会放大系统已有错误。

### 9.5 原则五: 指标推进必须分两条线

后续实验不能再把所有收益混在一起解释。

应该明确分成两条改进线:

#### 线 A: 身份连续性线

目标改善:

- `IDSW`
- `Frag`
- `IDs`
- `IDF1`
- `HOTA`

#### 线 B: 召回与误检线

目标改善:

- `FN`
- `FP`
- `MOTA`

若只做 A，不做 B:

- `MOTA 67` 基本无望

若只做 B，不做 A:

- `IDSW < 500` 也无望

## 11. 对未来方案边界的明确建议

2. 把恢复能力前移到 `lost` 阶段

4. 把 spatial prior 降级为可靠偏置项而不是强裁判


## 12. 最终结论

最终结论只保留最核心的几条。

### 12.1 方向层面

当前方案方向没有根本性错误。

固定监控场景下做:

- 生命周期特化
- 长时空间先验
- 高风险阶段外观恢复

这些方向都是成立的。

### 12.2 实现层面

当前版本存在两类真正致命的问题:

1. 状态机实现 bug
2. 恢复链路插入时机错误

这两类问题共同导致:

- 轨迹先被切碎
- 再很晚地、很苛刻地尝试补救
- 最终造成大量不必要的 ID 碎裂

### 12.3 设计层面

当前最大设计缺陷不是“模块太少”，而是:

- 模块很多
- 但没有形成统一控制体系

也就是说，问题不在于“还缺一个模块”，而在于“现有模块之间的关系不对”。

### 12.4 达标路径层面

如果要真正逼近:

- `HOTA 54`
- `MOTA 67`
- `IDF1 67`
- `IDSW < 500`

那么根本路径应该是:

1. 修状态机
2. 恢复前移
3. 重构 birth/exit/prior 控制逻辑
4. 然后才是进一步 tuning 和速度优化

如果不先做前两步，而继续围绕 zombie rescue 和参数微调反复试验，整体收益大概率会很有限。
