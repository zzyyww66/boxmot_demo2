# SOMPT22 不同数据集与检测器导致 IDSW 差异的原因分析 (2026-02-23)

本文档用于解释为什么“同一套改进版 ByteTrack”在不同数据集/不同检测器上，`IDSW` 改善幅度会差很多，并给出在 `YOLOv8m + SOMPT22` 上追求更大 `IDSW` 降幅的可行性判断与落地方向。

## 1. 你看到的现象 (用真实跑出的数值举例)

### 1.1 SOMPT22-full + YOLOX_x (本机最新一次全序列)

- 结果目录:
  - Baseline: `runs/sompt22_full_yoloxx_compare/mot/yolox_x_MOT17_ablation_lmbn_n_duke_bytetrack`
  - Improved: `runs/sompt22_full_yoloxx_compare/mot/yolox_x_MOT17_ablation_lmbn_n_duke_bytetrack_2`
- `IDSW`: `927 -> 750` (下降 `19.09%`)
- 但 `IDF1/HOTA` 有所下降，主要原因是 `CLR_FN` 增加 (召回下降)。

### 1.2 SOMPT22-full + YOLOv8m (本机 latest full-run: `runs/sompt22_compare_v8m_full_final`)

- 结果目录:
  - Baseline: `runs/sompt22_compare_v8m_full_final/mot/yolov8m_pretrain_crowdhuman_lmbn_n_duke_bytetrack`
  - Improved: `runs/sompt22_compare_v8m_full_final/mot/yolov8m_pretrain_crowdhuman_lmbn_n_duke_bytetrack_2`
- `IDSW`: `1117 -> 970` (下降约 `13.2%`)
- 在该组实验里，改进版对 `IDSW` 有帮助，但提升幅度明显小于你在 `MOT17-04` 上看到的 `-50%`。

### 1.3 旧机结果 (根目录 `SOMPT22测试/` 的对比报告)

`SOMPT22测试/BYTETRACK_COMPARISON_REPORT.txt` 报告显示 `IDSW` 下降 `33.8%`，但它与本机 `SOMPT22-full` 的结果不一定可直接对齐，常见差异来源包括:

- 数据集版本/序列帧数是否完全一致 (如 `12600` vs `14100` 帧)。
- 检测是否重跑、检测缓存是否一致 (不同 dets 文件会直接改变 `IDSW`)。
- baseline 的阈值是否是真正原版 ByteTrack (baseline 越“苛刻”，越可能压缩改进空间或改变相对提升比例)。

## 2. 为什么不同数据集/检测器会造成 IDSW 改善幅度差很多

核心结论: **`IDSW` 不是“只由跟踪器决定”的指标，它高度受检测质量与数据分布影响。**

### 2.1 检测器影响 (决定了“什么时候丢、丢成什么样”)

对 ByteTrack 这类纯运动关联方法来说，检测器差异会通过以下路径放大到 `IDSW`:

- **漏检 (FN)**: 目标断轨 -> 进入 `lost` -> 之后要么被错误重关联，要么被重新出生，都会增加 switch 风险。
- **框抖动/尺度抖动**: IoU 距离变差 -> 正确匹配失败 -> switch/fragment 增加。
- **重复框/近邻误检**: 在拥挤场景中把 A 的框分给 B -> 造成 switch。

不同权重 (如 MOT17 专训 vs CrowdHuman 预训练) 的抖动/FN/FP分布不同，`IDSW` 改善空间也会不同。

### 2.2 数据集影响 (决定了“你的先验是否成立”)

你提出的核心先验是:

> 监控画面中央不可能出现新目标，目标只会从边缘/入口进入；只有极少特殊情况或极度拥挤时才允许中央新生。

这个先验在不同数据集上成立程度不同:

- 在类似 `MOT17-04` 这种“固定监控视角 + 行人主要从边缘进入”的场景，**禁止中央新生**会直接减少“中央误出生导致的 IDs/IDSW”，因此可能出现 `IDSW -50%` 且其他指标不降反升。
- 在 `SOMPT22` 这类足球/运动场景里，**拥挤交叉、遮挡频繁、镜头变焦/平移更常见**，导致 switch 的主要来源更可能是“关联混淆”而不是“中央误出生”。此时仅靠 entry gate 很难拿到 MOT17-04 那种夸张的收益。

### 2.3 baseline 配置影响 (决定了“提升空间”)

如果 baseline 不是严格的“原版 ByteTrack 参数”，那么:

- baseline 的 `IDSW` 本身可能被人为压高或压低，导致改进幅度的百分比波动很大。
- 同一改进策略在不同 baseline 下，可能出现“看起来收益变小/变大”的错觉。

因此在做横向对比前，需要先把 baseline 固定到真正原版。

## 3. 为什么 YOLOv8m + SOMPT22 想做到 IDSW -40% 很难，但并非完全不可能

### 3.1 难点

要从 `1117 -> ~670` (约 `-40%`) 且要求 `HOTA/IDF1` 不掉甚至上升，本质上要求:

- 减少 switch 的同时
- 不增加 FN (不掉召回)
- 也不引入更多错误重关联 (不掉关联质量)

对纯运动 ByteTrack 来说，这在“拥挤交叉 + 遮挡 + 可能存在镜头运动”的数据上很难仅靠阈值实现。

### 3.2 可能的路线

更现实的方案通常是“把先验写进逻辑”，而不仅仅是调 `track_thresh/match_thresh`:

1. **中心禁止新生 + 可控例外机制**
   - 默认: 中心区域 `strict_entry_gate` 生效 (不允许直接新生)。
   - 例外 A (极少特殊情况): 中心检测连续稳定出现 `K` 帧才允许新生 (如 `K=3~5`)。
   - 例外 B (极度拥挤保召回): 当画面密度很高/未匹配高分检测持续过多时，临时放宽中心新生条件以控制 `FN`。

2. **加入相机运动补偿 (CMC)**
   - `SOMPT22` 中镜头运动会显著增加 IoU 匹配失败与错配。
   - 对这类场景，CMC 往往能在“不用 ReID”的前提下降低 switch/fragment。

3. **把 entry/exit 规则做成 ROI 而不是对称 margin**
   - `entry_margin` 是四边对称，不一定符合“道路入口只在某一侧/某几侧”的真实监控先验。
   - ROI/mask 更能把先验用对地方，减少误杀召回。

## 4. 推荐的落地实验顺序 (避免走弯路)

1. 先把 baseline 固定到“真正原版 ByteTrack 参数”(本仓库已完成: `boxmot/configs/trackers/bytetrack_original.yaml`)。
2. 在 `SOMPT22-08` 和 `SOMPT22-10` 上做小规模验证 (它们是 `IDSW` 贡献最大的序列)。
3. 如果目标是 `-40%` 且不掉 `IDF1/HOTA`，优先考虑实现 “中心禁止新生 + 可控例外” 和 “CMC” 两条逻辑改动，而不是无止境调阈值。
